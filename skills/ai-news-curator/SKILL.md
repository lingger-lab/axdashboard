# AI시대예감 뉴스 큐레이터 (스킬 1)

집출(zipchul.cloud)의 「AI시대예감」 발행 파이프라인 첫 단계.
Brad가 고를 수 있도록 **잘 거른 후보 3건 + 코멘트 발화점**을 차린다.

## 이 스킬의 정확한 범위

```
[스킬 1: 큐레이터]  뉴스 검색 → 후보 3건 선별 + 반론 제시   ← 지금 이 스킬
        ↓ Brad가 1건 선택 + 코멘트
[스킬 2: 빌더]     선택된 1건 → 발행 가능한 글            (ai-newsletter-builder)
        ↓
[스킬 3: 숏폼]     완성된 글 → 숏폼 패키지                (별도 스킬)
```

**입력:** "오늘 후보 뽑아줘" 류의 호출 (특정 키워드/카테고리 지정 가능)
**출력:** 후보 3건 (국내 2 + 해외 1), 각 건당 요약 + 출처 + "Brad라면 던질 반론·질문 3개"
**실행 방식:** Brad가 대화에서 호출할 때 실행. 매일 9시 자동 실행은 이 스킬이 아니라
  별도 인프라(크론+API)의 몫. 이 스킬은 "검색·선별하는 머리"만 담당한다.

## 핵심 원칙: 큐레이터는 최종 결정자가 아니라 1차 거름망

목표는 Brad의 판단을 복제하는 게 아니다. **버릴 80%를 걸러내고,
고를 만한 20%를 잘 차리는 것**이다. 최종 선택과 코멘트는 Brad가 한다.
그래서 후보는 "정답 1개"가 아니라 "고를 맛이 나는 3개"여야 한다.

## 타깃: AI로 창업·사업하려는 스타트업·중소기업·소상공인

연령·지역이 아니라 **사업 단계·규모**로 정의된 타깃.
판단 기준은 "4050이 체감하나"가 아니라 **"이 뉴스로 내일 뭘 해볼 수 있나"**.

## 선별 3대 질문 (모든 후보가 통과해야 함)

상세 기준·예시는 `references/selection-criteria.md` 참조.

```
Q1. 주제 경계 — AI로 '창업·사업하는 수익모델' 뉴스인가?
    ✅ "AI로 이렇게 매출 냈다"   ❌ "오픈AI 새 모델 출시"(기술 뉴스)

Q2. 관점 가능성 — Brad(경남 30년 기획자)가 한마디 보탤 각도가 있는가?
    객관적 중요도가 아니라 'Brad와의 화학반응'으로 거른다.

Q3. 타깃 적합성 — 스타트업·중소기업·소상공인이 '실행에 옮길 수 있는' 규모인가?
    ✅ 1인~소규모가 따라할 수 있는 것   ❌ 빅테크 수십억 투자, 과도한 기술
```

## 채널 역할 분리 (검색 우선순위: 유튜브 > 구글 > 다음)

| 채널 | 역할 | 무엇을 잡나 |
|------|------|------------|
| 유튜브 (1순위) | 사례·실전 | "AI로 이렇게 했다" 케이스. 가장 직접적 |
| 구글 (2순위) | 사실·출처 | 유튜브 사례의 진위 확인, 1차 출처 보강 |
| 다음 (보조) | 국내 반응 | 후보가 한국 창업 생태계에서 실제 논의되나 확인 |

> ⚠️ 다음은 '뽑는 곳'이 아니라 '검증하는 곳'. 유튜브·구글에서 뽑은 후보를
> 다음에서 재검색해 국내 회자 여부만 본다. 다음에서 새로 뽑지 않는다.

## 국내 2 : 해외 1 구성 (역할로 구분, 국적으로가 아니라)

```
국내 사례 2건 — "즉시 실행 증거"
  · 한국어 검색. "나도 당장" 동기를 주는 것 우선
  · 한국 시장·한국어·한국 결제로 이미 된 사례

해외 사례 1건 — "미래 신호" (와일드카드 슬롯)
  · 영어 검색. 6개월~1년 뒤 한국에 올 흐름
  · ★ 반드시 '한국 이식 가능성'이 보이는 것만. 한국에선 무리인 사례는 제외
    (안 그러면 Brad가 부정적 결론만 낼 수밖에 없는 뉴스가 됨)
  · Brad의 '번역가' 강점(이게 한국 오면 이렇게 변형된다)을 살리는 슬롯
```

## ★ 과장 필터 (유튜브 1순위의 최대 리스크)

유튜브 AI 수익 콘텐츠는 "월 천만원" "하루 만에" 낚시가 알고리즘 상위를 점한다.
조회수만 보고 뽑으면 후보 3건이 죄다 과장 사례가 된다.

```
후보 선정 시 각 건에 대해:
1. 수익 주장에 '근거'가 있나? (구체적 방법·기간·비용 공개 여부)
2. 근거 없는 과장("하루 만에 월천")은 와일드카드로도 뽑지 않는다
3. 뽑되 과장이 섞였으면 → 후보 메모에 "수치 미검증" 플래그를 단다
   (빌더가 '~라고 주장'으로 처리하도록 신호 전달)
```

## 워크플로우

```
[Step 1] 검색 범위 확인  — Brad가 카테고리/키워드 지정했나? 없으면 전체
[Step 2] 국내 검색       — 유튜브(한국어) → 구글 보강. 사례 다수 확보
[Step 3] 해외 검색       — 유튜브(영어) → 구글 보강. 한국 이식 가능한 것만
[Step 4] 과장 필터링     — 근거 없는 과장 제거, 미검증 수치 플래그
[Step 5] 다음 교차검증   — 후보의 국내 회자 여부 확인
[Step 6] 3건 선별        — 국내 2 + 해외 1, 카테고리 분산 고려
[Step 7] 반론 생성       — 각 건당 'Brad라면 던질 반론·질문 3개'  ★ 빌더 연결점
[Step 8] 출력            — references/output-format.md 템플릿
```

## ★ Step 7: 반론 생성 — 빌더와의 연결점

이게 큐레이터가 단순 뉴스 수집기와 다른 핵심이다.
각 후보에 "Brad라면 이 뉴스에 뭐라고 토를 달까"를 3개 미리 뽑아준다.
→ Brad는 0에서 코멘트를 쓰지 않고, 그중 하나에 *반응*하면 된다.
→ 그 반응이 곧 빌더(스킬 2)의 입력(코멘트 씨앗)이 된다.

반론 3개는 서로 다른 각도로:
- 반론 A (회의): "이거 그대로는 무리다" 류 — 숨은 비용·한계
- 반론 B (연결): Brad 경험/지역과 닿는 각도 — 경남·30년·현장
- 반론 C (질문): 독자에게 던질 물음 — "그럼 우리는 뭘?"

> 반론은 Brad의 입을 빌리지 않는다. '이런 각도가 있다'는 발화점 제안일 뿐,
> Brad가 실제 그렇게 생각한다고 단정하지 않는다.
