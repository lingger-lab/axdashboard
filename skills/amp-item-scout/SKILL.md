# AMP Item Scout v3 — Wide+Deep 수시스캔

## 3가지 모드

**수집 모드** — 30개 소스 전체 스캔 + 심층 검증 + DB 저장
- 트리거: "아이템 스카우트 해줘", "아이템 발굴", "DB 업데이트", "전체 스캔"

**매칭 모드** — 의뢰자 경력 → DB에서 맞는 아이템 제시
- 트리거: "내 경력에 맞는", "○○ 20년인데 뭘", "의뢰자 매칭"

**킬 테스트 모드** — 특정 아이템의 실패 가능성 심층 분석
- 트리거: "IT001 킬 테스트", "○○ 아이템 킬 테스트"

---

## 수집 모드: 3-Phase 실행

### ═══ Phase 1: WIDE (넓이) ═══
> 목표: 30개 소스를 7개 압축 검색으로 훑어 후보 10~15건 추출
> 소요: 검색 7회, 약 5분

**실행 순서:**

#### WIDE-1: 한국 실거래 (검색 2회)
```
검색A: "크몽 AI 인기 서비스 자동화 [현재연도]"
  → ① 크몽 실거래, 리뷰 수, 가격대
  
검색B: "탈잉 숨고 클래스101 AI 인기 [현재연도]"
  → ② 탈잉 ③ 숨고 ④ 클래스101
```
**추출 기준:** 리뷰 50건+, 거래량 상위, 신규 카테고리

#### WIDE-2: 한국 콘텐츠+매거진 (검색 2회)
```
검색C: "AI 1인 창업 부업 아이템 수익화 [현재연도]"
  → ⑤ 유튜브 ⑦ 블로그 ⑧ 브런치 + 자연 노출 소스

검색D: "동기부여학과 히든리치 조코딩 아이보스 캐릿 이오플라 AI 사업"
  → ⑥ 유튜브채널 ⑨ 아이보스 ⑩ 캐릿 ⑪ 이오플라
  → + ㉔ 히든리치스 ㉕ 솔비나 ㉖ 요즘IT ㉗ 세컨드샐러리
```

#### WIDE-3: 한국 불만/수요 (검색 1회)
```
검색E: "블라인드 네이버카페 업무 자동화 불만 AI 서비스 필요 [현재연도]"
  → ⑬ 네이버카페 ⑭ 블라인드 ⑮ GeekNews ⑫ 디스콰이엇
```
**추출 기준:** "이런 서비스 있었으면", "○○ 너무 불편하다" 패턴

#### WIDE-4: 해외 실매출 (검색 2회)
```
검색F: "indiehackers MRR AI solo founder [현재연도] revenue"
  → ⑯ Indie Hackers (실제 MRR이 핵심)

검색G: "producthunt trending AI tools small business theresanaiforthat appsumo [현재연도]"
  → ⑰ Product Hunt ⑱ TAAFT ⑲ AppSumo
  → + ⑳ MicroLaunch ㉑ SaaSHub ㉒ BetaList
```

#### WIDE 결과 정리
```
후보 리스트 10~15건 추출
각 후보에 대해:
  - 아이템명
  - 어디서 발견했나 (소스)
  - 왜 주목하는가 (1줄)
  - 깊이 파볼 가치 [상/중/하]

→ 상위 5건을 Phase 2로 넘긴다
```

---

### ═══ Phase 2: DEEP (깊이) ═══
> 목표: 상위 5건에 대해 실제 데이터로 시장성 4가지 검증
> 소요: 검색 3~5회 + web_fetch 2~3회, 약 5분

**각 아이템에 대해 아래 4가지를 검증:**

#### DEEP-1: 해외 팔리는 증거
```
검색: "[유사 서비스명] MRR revenue users"
또는: web_fetch로 Indie Hackers 게시물 직접 접근
목표: 실제 MRR, 이용자 수, 가격 구조
```

#### DEEP-2: 한국 팔리는 증거
```
검색: "크몽 [아이템 키워드] 서비스" 또는 "[한국 유사 서비스명]"
또는: web_fetch로 크몽 개별 서비스 페이지 접근
목표: 리뷰 수, 거래량, 가격대, 경쟁 서비스 수
```

#### DEEP-3: 불만 증거
```
검색: "블라인드 네이버카페 [고통 키워드] 불만 어렵다"
목표: 실제 사용자 목소리, 게시물 빈도
```

#### DEEP-4: 킬 리스크
```
검색: "[아이템 키워드] 실패 대체 무료 한계"
또는: "failory [유사 서비스] failure"
목표: 이 아이템이 안 될 이유 1~2가지
```

**DEEP 결과 → 아이템 카드 완성:**
```
각 아이템에 대해 전체 필드 채움:
  아이템명, 고객, 고통, AI 80%, 사람 20%,
  결과물, 예상가격, 필요역량키워드, 필요경력,
  해외증거, 한국증거, 불만증거, 킬리스크,
  MVP난이도, 데이터신뢰도, 소싱출처
```

**DEEP 품질 규칙:**
- web_fetch를 최소 2회 사용하여 실제 페이지 데이터 확인
- 숫자(리뷰 수, MRR, 이용자 수)가 있으면 반드시 포함
- 없는 데이터는 만들지 않는다 → "근거 없음 — 현장 인터뷰 필요"
- [실시간검색] / [추정] 태그 필수

---

### ═══ Phase 3: SAVE (저장) ═══
> 목표: 기존 DB와 대조 → 중복 제거 → 통합 DB 생성
> 소요: 약 2분

#### SAVE-1: 중복 대조
```
기존 DB(현재 52건)의 "아이템명"과 "한줄설명"을 대조
- 동일 아이템 → 기존 카드에 "최신 근거 업데이트" 추가
- 유사 아이템 → "⚠ 기존 IT0XX 유사" 표시 후 신규 등록
- 완전 신규 → 새 item_id 부여 (IT053~)
```

#### SAVE-2: DB 저장
```
우선순위:
1. Notion MCP 연결 시 → "AMP_아이템DB"에 직접 페이지 추가
2. Notion 미연결 → XLSX 파일 생성
3. 어떤 경우든 화면에 결과 출력
```

#### SAVE-3: 결과 출력
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    고통: "○○○"

    한 끗: ○○○
    ────
    해외: ○○○ [실시간검색] ← web_fetch 확인
```

## 매칭 모드

### STEP 1: 의뢰자 정보 수집
이미 제공된 경우 생략. 부족하면 물어본다:
```
1. "어떤 일을 몇 년 하셨나요?"
2. "가장 자신 있는 것은?"
3. "현장에서 나만 느끼는 감각이 있다면?"
```

### STEP 2: DB 검색
1. 의뢰자 입력에서 키워드 추출
2. Notion DB의 "필요역량키워드" 속성에서 검색
3. Notion 미연결 시 → 레퍼런스 파일 + 최신 XLSX에서 검색

### STEP 3: 매칭 실패 시 폴백
DB에 매칭 없으면:
1. "현재 DB에 [○○] 관련 아이템이 없습니다"
2. **즉석 웹 검색**으로 3건 제안 (크몽 + 유튜브 + IH)
3. "DB에 등록할까요?" → 승인 시 추가

### STEP 4: 매칭 결과 출력
```
━━━━━━━━━━━━━━━━━━━━━━━━━━
  [의뢰자 경력]에 맞는 아이템
━━━━━━━━━━━━━━━━━━━━━━━━━━

입력: ○○ ○년, ○○ 특기

[1] 아이템명
    고객: ○○○
    고통: "○○○"
    AI 80%: ○○○
    당신의 한 끗: ○○○
    ─────
    해외: ○○○
    한국: ○○○
    예상가격: ○○○

→ Brad와 함께 수익 구조를 설계합니다.
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 집중 스캔 (소스 지정)

사용자가 특정 소스를 지정하면 해당 채널만 집중 스캔:
```
"크몽만 봐줘" → WIDE-1의 검색A만 + DEEP 전체
"해외 소스만" → WIDE-4 + DEEP 전체  
"불만 채널 스캔" → WIDE-3 + DEEP-3 집중
"히든리치스 최신" → web_fetch로 직접 접근 + DEEP
```

## 주의사항

- 대형 플랫폼 자체(Zapier, Notion 등)를 아이템으로 수집하지 않는다
- Product Hunt 결과가 대형 도구 위주이면 건너뛰고 크몽/유튜브로 간다
- 한국 시장 근거가 더 중요하다. 해외 MRR은 참고용
- "해외에서 통한 것 = 한국에서도 통한다"고 가정하지 않는다
- 킬 리스크를 반드시 포함한다. 낙관 편향을 방지한다
- Phase 2(DEEP)에서 web_fetch를 반드시 사용한다 — 검색 결과 스니펫만으로는 깊이가 부족
- 인디해커 벤치마크: 첫 매출 평균 3개월, $1K MRR 평균 6~18개월
- "도구를 판다"가 아니라 "결과를 판다"(outcome-as-a-service)로 아이템을 정의한다
