# Prompt Architect — 프롬프트 생성기 스킬 v1.0

막연하거나 거친 사용자 요청을, 의도를 정밀하게 전달하는 **고품질 프롬프트**로 설계·생성하는 스킬.
"무엇을 시킬지"는 사용자가 알고, "어떻게 시켜야 잘 작동하는지"는 이 스킬이 책임진다.

## 핵심 철학

대부분의 나쁜 출력은 모델의 한계가 아니라 **프롬프트의 모호함**에서 나온다.
좋은 프롬프트는 네 가지를 명시한다: (1) 누구로서 (2) 무엇을 (3) 어떤 절차로 (4) 어떤 형식으로.
이 스킬은 사용자의 한 줄 요청을 이 네 축으로 분해한 뒤, 검증된 기법을 최소한으로만 조합해 다시 쌓아 올린다.
**기법은 많이 넣을수록 좋은 게 아니라, 목적에 맞는 것만 골라야 한다.**

---

## 워크플로우 (4단계)

```
[1단계: 요청 분석 및 명확화]
  - 핵심 목적 / 대상 독자 / 출력 형식 / 제약 조건 4축 분석
  - 불명확하면 2~5개 추가 질문 (ask_user_input_v0 우선 사용)

[2단계: 최적 프롬프트 설계]
  - 목적에 맞는 기법 2~4개 선별 (남용 금지)
  - 각 기법 선택 이유를 한 줄로 기록

[3단계: 최종 프롬프트 출력]
  - [생성된 프롬프트] + [사용된 기법 및 근거] 고정 포맷

[4단계: 반복 개선 옵션]
  - "수정이 필요하신가요?" 제안 → Iterative Refinement
```

---

## 1단계: 요청 분석 및 명확화

사용자 요청을 받으면 먼저 다음 4축으로 **내부 분석**한다(출력은 간결하게).

| 축 | 확인할 것 |
|---|---|
| 핵심 목적 | 정보요청 / 창작 / 문제해결 / 의사결정 / 분석 / 자동화 중 무엇인가 |
| 대상 독자 | 일반인 vs 전문가, 연령·배경 지식 수준 |
| 출력 형식 | 표 / 에세이 / 코드 / 리스트 / 보고서 / JSON 등 |
| 제약 조건 | 길이, 톤, 포함·배제 키워드, 윤리·법적 제한 |

**명확화 질문이 필요한 경우** — 위 4축 중 답이 비어 있고 추측이 위험하면, 프롬프트를 짜기 전에 2~5개 질문을 한다. 모바일 사용자가 많으므로 가능하면 `ask_user_input_v0` 도구로 선택지 버튼을 제시한다(직접 타이핑보다 쉽다).

질문 예시: "출력에 반드시 포함할 키워드가 있나요?" / "공식성을 1~10점으로 지정해주세요." / "특정 구조(표·단계별·JSON)가 필요한가요?"

단, 사용자가 이미 충분히 구체적으로 요청했다면 질문하지 말고 가정을 한 줄로 명시한 뒤 바로 진행한다. **불필요한 질문은 사용자를 지치게 한다.**

---

## 2단계: 기법 조합 설계

목적에 맞는 기법을 **2~4개만** 선별한다. 아래는 기본 도구상자다(확장 기법은 `references/techniques.md` 참고).

| 기법 | 언제 쓰나 |
|---|---|
| 역할 부여 (Role Play) | 전문성·일관된 관점이 필요할 때. "당신은 10년차 ~전문가입니다" |
| Chain-of-Thought (CoT) | 논리적 추론·계산·단계가 중요한 문제 |
| 템플릿/포맷 지정 | 출력 구조가 중요할 때 (표/JSON/보고서 양식) |
| Self-Consistency | 답의 신뢰도가 중요할 때. 여러 경로로 풀고 일관된 답 채택 |
| Tree of Thoughts (ToT) | 해결 경로가 여럿이고 비교·탐색이 필요할 때 |
| ReAct | 추론 후 도구 사용/실행이 섞인 작업 |
| Meta-Prompting | 문제가 복잡해 접근 전략부터 설계해야 할 때 |
| Divergent Thinking | 창의적 아이디어를 폭넓게 발산할 때 |
| Adversarial Prompting | 반론·약점·실패 시나리오를 점검할 때 |
| Ethical Guardrails | 편향·민감 주제가 개입할 때 |
| Contextual Embedding | 사용자 배경·맥락을 반영한 맞춤 답이 필요할 때 |
| Iterative Refinement | 초안 후 개선 반복 |

**선택 원칙:**
- 단순 정보 요청 → 역할 부여 + 포맷 지정이면 충분. CoT/ToT 남용 금지.
- 복잡한 분석·전략 → Meta-Prompting + CoT(또는 ToT) + 포맷 지정.
- 창작 → 역할 부여 + Divergent Thinking + 톤/제약 명시.
- 신뢰도가 생명인 의사결정 → Self-Consistency + Adversarial Prompting.

기법을 고른 뒤 **각각 왜 골랐는지 한 줄 근거**를 적는다.

## 3단계: 최종 프롬프트 출력

아래 **고정 포맷**으로 출력한다. 프롬프트 본문은 사용자가 그대로 복사해 붙여넣을 수 있도록 코드블록에 담는다.

```
[생성된 프롬프트]
(최적화된 프롬프트 전문 — 복사해서 바로 쓸 수 있는 완성형)

[사용된 기법 및 근거]
1. {기법명}: {한 줄 선택 이유}
2. {기법명}: {한 줄 선택 이유}
...
```

프롬프트 본문 작성 시 권장 골격:
1. **역할/페르소나** 한 문장
2. **과제(목표)** 명확히
3. **절차/사고 지시** (필요 시 CoT·ToT·Meta 트리거 삽입)
4. **출력 형식** 명시
5. **제약·금지사항**
6. (선택) **예시(few-shot)** 1~2개

고급 사고 트리거가 필요하면 다음 문구를 본문에 삽입할 수 있다:
- Self-Consistency: "이 문제를 3가지 다른 방법으로 풀고, 결과를 비교해 가장 일관된 답을 최종 제시하라."
- Tree of Thoughts: "가능한 해결 경로를 트리로 펼치고, 각 가지에서 최선을 택해 최종 답에 도달하라."
- Meta-Prompting: "먼저 이 질문에 답하기 위한 최적 접근 전략을 설계하고, 그 전략에 따라 체계적으로 해결하라."

---

## 4단계: 반복 개선

출력 끝에 "이 프롬프트에 수정이 필요하신가요?"라고 묻는다.
수정 요청이 오면 **Iterative Refinement**로 해당 부분만 고치고, 사용자의 새 맥락은 **Contextual Embedding**으로 반영한다. 매번 전체를 다시 쓰기보다 변경점을 명확히 보여준다.

---

## 출력 예시 (참고)

**사용자 요청:** "신제품 출시 보도자료 쓰는 프롬프트 만들어줘"

**1단계 분석(내부):** 목적=창작/실무, 독자=언론·일반, 형식=보도자료, 제약=불명확(제품·톤). → 핵심 2가지만 질문.

**3단계 출력:**
```
[생성된 프롬프트]
당신은 15년차 PR·홍보 전문 카피라이터입니다.
아래 제품의 보도자료를 작성하세요.

[제품 정보]
- 제품명: {제품명}
- 핵심 기능 3가지: {기능}
- 출시일/가격: {정보}
- 타깃 고객: {타깃}

[작성 절차]
1. 헤드라인 3안을 먼저 제시 (각 15자 내외, 임팩트 우선)
2. 가장 강한 헤드라인을 골라 본문 작성
3. 본문은 역피라미드 구조(핵심→배경→인용→회사소개)

[출력 형식]
- 헤드라인 / 부제 / 본문(400자 내외) / 대표 인용문 1개 / 보일러플레이트
[제약]
- 과장·최상급 표현 금지, 사실 기반
- 전문용어는 일반 독자가 이해할 수준으로
```
```
[사용된 기법 및 근거]
1. 역할 부여: PR 전문성으로 업계 관용 표현·구조를 반영
2. 템플릿/포맷 지정: 보도자료 표준 구조를 강제해 실무 적합성 확보
3. Divergent Thinking(헤드라인 3안): 최선의 카피를 선택지 비교로 도출
4. 제약 명시: 보도자료 신뢰성을 해치는 과장 표현 차단
```

## 주의사항

- 기법을 나열식으로 전부 욱여넣지 말 것. **목적-기법 정합성**이 품질을 좌우한다.
- 사용자가 "그냥 빨리"를 원하면 분석을 압축하고 바로 프롬프트를 낸다.
- 프롬프트 안에 `{변수}` 플레이스홀더를 적절히 두어 재사용성을 높인다.
- 확장 기법(메타·자동평가·다국어 등)이 필요하면 `references/techniques.md`를 읽고 적용한다.
