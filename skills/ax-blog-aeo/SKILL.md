# AX 블로그 AEO 변환기 (스킬 3 · ② 생성 > 발견)

완성된 글을 **AI가 인용하는 형태**로 변환한다. 퍼널에서 잠재고객의 **발견**을 담당 —
사장님이 ChatGPT에 "직원 안 뽑고 버티는 법" 물었을 때 Brad 글이 답으로 나오게 한다.

## 경계: 이 스킬은 '변환'만 한다

```
[스킬 2 or 4]  완성된 글 (사람이 읽는 형태)
        ↓
[스킬 3: AEO]  같은 글 → AI가 인용하는 구조        ← 지금 이 스킬
              (내용 불변, 구조만 변환)
```

**입력:** 완성된 글 1건 + 4기둥 태그
**출력:** AEO 블로그 글 (질문형 제목 + 결론선두 + 본문 + FAQ schema 마크업 + 메타)

> ⚠️ 콘텐츠를 두 번 만들지 않는다. 이 스킬은 *재구조화*지 *재창작*이 아니다.
> Brad 코멘트의 내용은 절대 손대지 않는다 — 감싸는 구조만 바꾼다.

## 뉴스레터 vs AEO 블로그 — 변환 규칙표

| 요소 | 원본 (사람용) | 변환 후 (AI 인용용) |
|---|---|---|
| 제목 | 후킹형 | **질문형 + 4기둥 고객언어** |
| 첫 문단 | 도입·분위기 | **결론 먼저** (40~60자 직답) |
| 소제목 | 자유 | **질문형으로** |
| Brad 코멘트 | 서사 흐름 | **인용 가능한 단문으로 분리** (내용 불변) |
| 끝단 | CTA | **FAQ 블록 + schema 마크업** |

## 4기둥별 질문형 키워드 세트 (이 스킬의 자산)

검증 단계 = **인건비 기둥 10개에 집중.** 나머지는 인용 증거 후 확장.

인건비절감 기둥 질문 예시:
- "직원 안 뽑고 버티는 방법은?"
- "1인 사장이 AI로 직원 한 명 몫 하려면?"
- "최저임금 오르는데 사람 대신 뭘 쓰지?"
- "소상공인 인건비 줄이는 현실적인 방법?"
(→ 이 질문들이 곧 제목·소제목·FAQ 질문의 원천)

## 워크플로우

```
[Phase 0] 입력 확인        — 완성된 글 + 4기둥 태그
[Phase 1] 제목 질문형화     — 4기둥 고객언어 질문으로
[Phase 2] 결론 선두 추출    — 핵심 답 40~60자를 첫 문단에
[Phase 3] 본문 재구조화     — 소제목 질문화, 단락 인용단위로
[Phase 4] Brad 코멘트 분리  — 인용 가능한 단문 블록으로 (내용 불변)
[Phase 5] FAQ schema 생성   — 질문 3~5개 + JSON-LD 마크업
[Phase 6] 발행 메타         — 슬러그·메타설명·OG (워드프레스용)
```

## Phase 2: 결론 선두 (AEO 핵심)

AI는 글 상단 요약을 인용한다. 첫 문단에 그 글의 *답*을 40~60자로 박는다.
예: "직원을 안 뽑고 버티려면, 반복 응대·예약·정산 세 가지를 먼저 자동화한다."
이게 Perplexity가 그대로 따올 문장이다.

## Phase 5: FAQ schema (JSON-LD)

질문형 키워드 세트에서 3~5개를 뽑아 FAQPage 스키마로. 검색엔진·AI가 구조를 인식한다.
검증 단계엔 손으로 삽입해도 무방 — 구조가 인용되는지부터 확인이 우선.

## ⚠️ AEO 과최적화 경고

AEO를 좇다 글이 질문-답-FAQ로만 수렴하면, AI엔 인용되는데 사람은 안 읽는 빈 껍데기가
된다. 목적은 "인용"이 아니라 "발견→전환"이다. Brad의 4,500 페북망은 사람이다.
**인용 가능성과 사람의 가독성을 둘 다 지킨다** — FAQ는 본문 뒤 보조일 뿐, 본문 서사는 산다.

## 검증 측정 (이 스킬의 합격 기준)

발행 2~3주 후: 질문형 키워드를 Perplexity·ChatGPT에 입력 → Brad 글 인용되나 확인.
선행지표(GSC 노출·스키마 인식) / 후행지표(실제 인용) 분리. + 인용 타고 온 유입이
오픈채팅까지 오는가(전환). 인용만으론 허영지표일 수 있다.
